## Part 1: Zero-Knowledge Proof (ZKP) Foundations

### 1. Protocol Architecture & Implementation
To prove knowledge of a 4-digit PIN $P$ without revealing the digits to the verifier or an eavesdropper, this project implements a **Commit-Challenge-Response** interactive protocol simulated over 100 iterations.

- **Commitment Phase**: The Prover selects a random 16-bit blinding factor $r_i$ and generates two hashes:
  - $C_{i, 0} = \text{SHA256}(r_i \mathbin{\Vert} \text{salt}_{i, 0})$
  - $C_{i, 1} = \text{SHA256}((P \oplus r_i) \mathbin{\Vert} \text{salt}_{i, 1})$
  Here, $\oplus$ is the bitwise XOR operation, acting as a secure one-time pad. The Prover sends $(C_{i,0}, C_{i,1})$ and their salts to the Verifier.
- **Challenge Phase**: The Verifier issues a random challenge bit $c_i \in \{0, 1\}$.
- **Response Phase**: 
  - If $c_i = 0$: Prover opens the left envelope, revealing $r_i$ and $\text{salt}_{i, 0}$.
  - If $c_i = 1$: Prover opens the right envelope, revealing $S_i = P \oplus r_i$ and $\text{salt}_{i, 1}$.
- **Verification**: 
  - If $c_i = 0$: Verifier checks $\text{SHA256}(r_i \mathbin{\Vert} \text{salt}_{i,0}) == C_{i,0}$.
  - If $c_i = 1$: Verifier checks $\text{SHA256}(S_i \mathbin{\Vert} \text{salt}_{i,1}) == C_{i,1}$ and reconstructs $r'_i = S_i \oplus P$ using their registered PIN. They confirm that $\text{SHA256}(r'_i \mathbin{\Vert} \text{salt}_{i,0}) == C_{i,0}$.

### 2. Academic Explanation: Why is this "Zero-Knowledge"?
In a Zero-Knowledge Proof, the Verifier must learn **zero** information about the secret except for the truth of the statement (that the Prover knows the PIN).

#### What exactly does the Verifier learn?
During any single iteration $i$, the Verifier learns either $r_i$ (a random 16-bit integer) or $S_i = P \oplus r_i$. 
- Since $r_i$ is chosen uniformly at random for each round, $r_i$ contains zero information about the PIN $P$.
- The value $S_i = P \oplus r_i$ is a ciphertext under a **One-Time Pad** (OTP). In information theory, an OTP provides **perfect secrecy** as long as the key ($r_i$) is random, used only once, and kept secret.
- Since the Verifier only gets one of the two pieces of information, they can never compute $r_i \oplus S_i = P$. 
- Consequently, the transcripts generated by the protocol look completely random, and the Verifier learns absolutely nothing about the PIN digits.

#### Soundness & Cheating Prover Success Probability
If a Prover does not know the PIN $P$, they can attempt to cheat by predicting the challenge $c_i$:
- If they guess $c_i = 0$, they commit to a valid random $r_i$ and a dummy $C_{i,1}$. If challenged with 0, they pass. If challenged with 1, they fail.
- If they guess $c_i = 1$, they commit to a valid $s_i$ and a dummy $C_{i,0}$. If challenged with 1, they pass. If challenged with 0, they fail.

The probability of cheating successfully on a single round is exactly $0.5$ (50%). 
For $n = 100$ independent iterations, the probability that a cheating prover succeeds in passing all challenges by random guessing is:
$$P(\text{cheat}) = \left(\frac{1}{2}\right)^{100} \approx 7.888 \times 10^{-31}$$
This is cryptographically negligible, providing the verifier with near $100\%$ statistical confidence.

## Part 2: Merkle Trees & Anonymity Sets

### 1. Anonymity Pool Simulation
To hide transaction histories, we model a deposit/withdrawal privacy pool:
- **Deposits**: 10 users deposit a "coin" (unique identifier string). The system computes the SHA-256 hash of each coin and places them as the first 10 leaves in a fixed-depth binary Merkle Tree (depth 4, capacity 16 leaves).
- **Placeholders**: The remaining 6 leaf slots are initialized with a standard `EMPTY_HASH`.
- **Root**: The root is computed by pair-wise hashing level-by-level to the top.

### 2. Path Proof & Verification
- `generateProof(leafIndex)`: Traces the tree from the specified leaf to the root and retrieves the sibling hashes and their positions (`is_left` flag).
- `verifyProof(leaf, proof, root)`: Takes only the raw coin string, hashes it to form the starting node, and ascends the tree using the sibling hashes. It returns `True` if the final hash matches the root. Note that **leafIndex is not passed to the verifier**.

### 3. Critical Questions (Written Deliverable)

#### Q1: If an outsider sees only the root and a valid proof, can they determine which leaf was spent? Why or why not?
**Answer**: 
It depends on whether the proof is presented in **plaintext** or inside a **Zero-Knowledge envelope**.

1. **If the raw Merkle proof is visible in plaintext**: 
   **Yes**, the outsider *can* determine exactly which leaf index was spent. The Merkle proof contains the sibling directions (`is_left` or `is_right`) for each layer of the tree. The sequence of left/right directions from the leaf to the root acts as a binary representation of the leaf's index. For example, in a tree of depth 4, a path requiring `[Right, Left, Right, Left]` corresponds directly to index $5$ (binary `0101`).
2. **If the proof is verified within a Zero-Knowledge Proof (like zk-SNARKs)**:
   **No**, the outsider cannot determine which leaf was spent. In privacy mixers like Tornado Cash, the Merkle path and the leaf index are treated as private witnesses. The prover generates a ZKP stating: *"I know a secret leaf and a valid Merkle path that hashes to the public root, and its nullifier hash is $H(n)$."* The outsider and the smart contract only see the public inputs (the root and the nullifier hash) and the cryptographic proof, which mathematically hides the Merkle path, making the spent leaf unlinkable.

#### Q2: What is the "anonymity set" size here?
**Answer**:
The anonymity set size is **10**.
An anonymity set represents the group of users among whom a given transaction is indistinguishable. Since there are 10 unique deposits inside the Merkle Tree, any verified withdrawal cannot be mapped to any specific depositor, meaning the spent leaf could be any of the 10 deposited coins. Thus, the anonymity set size is 10.
